# 月度事件前后端接口与请求事件入口

本文记录 `AutoMonthlyEvent` / `AutoMonthlyEvent.Executor` 当前确认过的月度事件接口、数据格式和请求事件分类。结论以源码定位为准；未找到调用方的入口单独标为“待继续追踪”，避免把观察结果误当成可靠 API。

## 1. 总体流程

月度事件的大致链路是：

1. 后端月度结算或 AI 行为调用 `MonthlyEventCollection.Add...`，把事件写入月度事件集合。
2. 游戏前端收到事件展示数据，`EventModel.OnNotifyGameData(...)` 反序列化并更新 `EventModel.DisplayingEventData`。
3. 事件窗口根据 `TaiwuEventDisplayData.EventOptionInfos` 展示选项。
4. 玩家或 MOD 调用 `EventModel.Select(optionKey)` / `TaiwuEventDomainMethod.Call.EventSelect(eventGuid, optionKey)` 推进事件。
5. 后端事件包按 `optionKey` 执行结果，可能再生成后续结果窗口、人生经历、情报记录或下一级 UI。

当前 Executor 最可靠的自动处理层是第 2-4 步：前端看到完整事件文本、角色和选项后，按白名单与条件选择选项。第 1 步适合做“截获事件生成”，但必须逐个确认具体后端入口，不能只 patch 一个 `RequestItemAction` 就认为覆盖全部请求。

## 2. 前端接口

### 2.1 事件模型入口

- 类：`EventModel`
- 文件：`spied/The Scroll of Taiwu/Assembly-CSharp_da99b9b8/EventModel.cs`
- 方法：`OnNotifyGameData(List<NotificationWrapper> notifications)`
- 关键字段：`DisplayingEventData`

`OnNotifyGameData` 中会从通知数据反序列化 `TaiwuEventDisplayData`。如果 `EventGuid` 为空，前端会把 `DisplayingEventData` 置空。

### 2.2 事件选择接口

- 前端方法：`EventModel.Select(string optionKey)`
- 后端调用包装：
  - `TaiwuEventDomainMethod.Call.EventSelect(string eventGuid, string optionKey)`
  - `TaiwuEventDomainMethod.Call.EventSelect(string eventGuid, string optionKey, bool isContinue)`

`EventModel.Select(optionKey)` 会检查当前 `DisplayingEventData.EventOptionInfos` 中是否存在该 `optionKey`，然后调用 `EventSelect(eventGuid, optionKey)`。这是前端自动点击的稳定入口。

### 2.3 事件窗口 UI 入口

当前源码中至少存在以下事件窗口类名：

- `UI_EventWindow`
- `Game.Views.EventWindow.ViewEventWindow`

关键私有方法：

- `SelectOptionByOptionKey(string optionKey)`
- `SelectOption(EventOptionInfo optionInfo)`

`SelectOptionByOptionKey` 会在当前 `Data.EventOptionInfos` 中查找目标 `OptionKey` 并调用 `SelectOption`。用 Harmony patch 时应同时尝试两个窗口类，并记录实际安装结果；只 patch 第一个找到的类可能错过游戏实际使用的窗口。

## 3. 前端数据格式

### 3.1 `TaiwuEventDisplayData`

- 文件：`GameData.Shared_e85ad006/GameData/Domains/TaiwuEvent/DisplayEvent/TaiwuEventDisplayData.cs`
- 常用字段：
  - `EventGuid`：事件 GUID。
  - `EventContent`：当前窗口文本，已经经过事件系统生成。
  - `MainCharacter`：左/主角色显示数据。
  - `TargetCharacter`：右/目标角色显示数据。
  - `EventOptionInfos`：当前可见选项列表。
  - `ExtraData`：物品选择、人物选择、输入请求等特殊 UI 数据。

自动处理需要至少记录：

- `eventGuid`
- `candidateType`
- `mainCharacterId`
- `targetCharacterId`
- `eventContent`
- `options[]`
- 事件签名：`eventGuid + optionKeys + 角色 ID + 文本 hash`

### 3.2 `EventOptionInfo`

- 文件：`GameData.Shared_e85ad006/GameData/Domains/TaiwuEvent/DisplayEvent/EventOptionInfo.cs`
- 常用字段：
  - `OptionKey`：提交给 `EventSelect` 的选项 key。
  - `OptionContent`：选项文本。
  - `OptionState`：选项状态。
  - `Behavior`：选项关联立场或行为显示语义，适合辅助分类，不应单独作为执行依据。
  - `OptionType`：选项类型。

当前 Executor 采用保守规则：

- `OptionState == 0` 才视为可自动点击。
- 请求类必须同时满足事件白名单、选项文案、选项状态与关系/好感条件。
- 未知事件和未命中白名单事件交给玩家。

## 4. 后端月度事件集合入口

### 4.1 可靠入口：`MonthlyEventCollection`

- 类：`GameData.Domains.World.MonthlyEvent.MonthlyEventCollection`
- 文件：`GameData.Shared_e85ad006/GameData/Domains/World/MonthlyEvent/MonthlyEventCollection.cs`

月度事件入队函数使用统一模式：

```csharp
int beginOffset = BeginAddingRecord(templateId);
AppendCharacter(...);
AppendLocation(...);
AppendItemKey(...);
AppendInteger(...);
EndAddingRecord(beginOffset);
```

因此，`templateId` 是静态分类最可靠的字段；运行时 GUID 则用于前端窗口匹配。

## 6. 结果记录层：`SecretInformationCollection`

- 类：`GameData.Domains.Information.Collection.SecretInformationCollection`
- 文件：`GameData.Shared_e85ad006/GameData/Domains/Information/Collection/SecretInformationCollection.cs`

该层提供 `AddAcceptRequest...` 和 `AddRefuseRequest...` 系列方法，例如：

- `AddAcceptRequestResource(int charId, int charId1, sbyte resourceType)`
- `AddRefuseRequestResource(int charId, int charId1, sbyte resourceType)`
- `AddAcceptRequestItem(int charId, int charId1, ulong itemKey)`
- `AddRefuseRequestItem(int charId, int charId1, ulong itemKey)`

这些方法用于记录接受/拒绝结果，不是请求事件的生成入口。它们适合用来验证后端结算是否发生，不适合直接作为自动处理入口。

## 7. 胎教与生育相关入口

### 7.1 胎教月度事件

- 方法：`MonthlyEventCollection.AddPrenatalEducationTaiwu(int charId)`
- templateId：`13`
- 记录格式：`charId`

当前 Executor 采用前端处理：识别胎教事件 GUID 后选择配置的第 1/2/3 个选项。后端“选项效果”还没有完全定位，暂不直接在后端模拟选项结果。

### 7.2 春宵事件

- 方法：`MonthlyEventCollection.AddMakeLoveWithTaiwu(...)`
- 当前后端实验 patch 能观察/阻止该事件加入集合。
- 是否应默认处理：不应默认开启。该类应放入“亲密/生育事件”或“后端实验截获”，由玩家明确开启。

## 8. 当前待继续追踪的问题

1. `MonthlyEventCollection.AddRequest...` 的上游调用方在当前反编译源码中未通过普通文本搜索定位到。可能原因是调用由生成器、事件包脚本或未反编译的逻辑间接触发。
2. 之前 patch 的 `RequestItemAction.ApplyInitialChangesForTaiwu` 只覆盖特定 AI 行为路径；实测资源请求没有命中该 patch，不能作为请求 66-86 的通用入口。
3. 后端直接处理请求需要确认两个额外问题：
   - 接受/拒绝选项最终调用哪些事件包动作。
   - 后续结果窗口是否必须生成，还是可以由后端直接结算并静默跳过。
4. 前端 `OptionKey` 需要继续通过 Discovery 采样确认稳定性。即使 `OptionKey` 稳定，执行前仍应复核 `OptionContent` 与 `OptionState`。

## 9. Executor 推荐实现分层

### 9.1 稳定层：前端白名单自动选择

适合当前版本默认使用：

- 请求系列 66-86：按 `templateId/GUID + 选项文案 + optionState + 关系/好感` 判断。
- 胎教：按配置选择 1/2/3。
- 明确白名单结果窗口：只在已识别类型下自动继续。

### 9.2 实验层：后端截获

适合逐项验证，不应总开关自动开启：

- 春宵：可在 `AddMakeLoveWithTaiwu` 层阻止加入月度集合。
- 物品请求：`RequestItemAction` 只作为特定路径，不代表 66-86 全覆盖。
- 赠礼：`GiveItemAction` 属于普通交互路径，不应混入月度请求默认逻辑。

### 9.3 不处理层

默认交给玩家：

- 切磋、挑战、较艺、比武、促织战斗。
- 奇遇、主线、剧情事件。
- 非白名单 normal event。
- 任何角色、选项、关系、时序信息缺失的事件。
